## 1. Purpose and Scope

### 1.1 Purpose

This Software Failure Mode and Effects Analysis (SW-FMEA) systematically identifies failure modes of the `scorehsm` Rust cryptographic library, their local effects at the API boundary, their potential system-level safety consequences, and the design mitigations and test coverage that reduce or eliminate those risks. It is produced in accordance with ISO 26262-6:2018 §7.4.9 (software architecture design verification) and the FMEA methodology described in ISO 26262-9:2018.

This document complements the threat model (`docs/safety/threat-model.md`), which addresses deliberate adversarial attacks. The SW-FMEA addresses **systematic and random software faults** — incorrect implementation, missed error conditions, type confusion, and logic errors — rather than attack vectors.

### 1.2 Scope

The following modules and functions are analyzed:

- Key management: `key_generate`, `key_import`, `key_delete`
- Symmetric authenticated encryption: `aes_gcm_encrypt`, `aes_gcm_decrypt`
- Asymmetric cryptography: `ecdsa_sign`, `ecdsa_verify`, `ecdh_agree`
- MAC: `hmac_sha256`
- Key derivation: `key_derive`
- Random generation: `random`
- Session and access control: `HsmSession` layer
- Application-level verification: `verify_update_image`, `verify_activation_token`

### 1.3 Severity Classification

Severity is assigned per ISO 26262 Part 3 (HARA) definitions, applied to `scorehsm` as a SEooC whose safety-relevant service is cryptographic authentication of safety-critical data:

| Level | ISO 26262 Definition | Application to scorehsm |
|---|---|---|
| S0 | No injuries | No realistic path from this failure to a vehicle hazard |
| S1 | Light and moderate injuries (reversible) | Incorrect non-safety data accepted; detectable by higher layers |
| S2 | Severe and life-threatening injuries (survival probable) | Incorrect safety-related message authenticated; fault detectable by system diagnostics with reasonable probability |
| S3 | Life-threatening (survival uncertain) or fatal | Malicious or corrupted firmware authenticated as valid → vehicle control software executes corrupted code; or valid safety message rejected → system fails to safe state with potential loss of vehicle control |

**S3 rationale for a cryptographic library:** `scorehsm` is the trust root for OTA firmware verification and inter-ECU message authentication in the target SDV platform. A failure that causes the library to accept an invalid signature (authentication bypass) can result in: (1) installation of attacker-controlled or corrupted firmware in a vehicle control ECU, (2) acceptance of spoofed safety-critical messages (e.g., brake command). Either outcome creates a direct path to an S3 vehicle hazard. Conversely, a failure that causes systematic rejection of valid firmware (false rejection) puts the vehicle in a state where safety updates cannot be applied, which is classified S1–S2 depending on the severity of the deferred update.

### 1.4 Residual Risk Definition

Residual risk is what remains after all documented mitigations are applied. It captures:
- Any failure mode not fully covered by design mitigations
- Any test gap that leaves a failure mode undetected before integration
- Any assumed safety requirement (ASR) on which the mitigation depends

## 2. FMEA Table

> Column definitions:
> - **FM-ID**: Failure mode identifier for traceability
> - **Function**: The API function or module where the failure originates
> - **Failure Mode**: How the function fails at its output boundary (observable incorrect behavior, not the root cause)
> - **Local Effect**: Immediate consequence at the API surface — what the caller receives
> - **System Effect**: Consequence if the failure propagates unchecked to the vehicle-level integrating system
> - **S**: ISO 26262 Severity (S0–S3)
> - **Mitigation**: Design, language enforcement, or architectural control that prevents or detects this failure
> - **Test**: Test cases or test categories that exercise this failure mode
> - **Residual Risk**: Remaining risk after mitigations
